A note on keeping what we know

On wikis
that don't rot

Most knowledge bases die the same quiet death: written once, read never, trusted by no one. We think the cure is to make a wiki worth two readers at once — a person, and the machine working beside them.

By the Aveline authors — Arie & Trevor
§
CHAPTER I

The rot is not the writing.
It's the forgetting.

Every team starts the same way — with good intentions and an empty page. Someone writes down how the deploy works, where the secrets live, why that one service is named after a cat. For a week it is true. Then the code moves on, and the page does not. The document was never wrong; it simply stopped being maintained the moment it was most needed.

in the margin We've watched four-year-old runbooks send brand-new engineers straight into walls. The page wasn't lying. It was just old.

The usual fix is discipline: review cycles, doc owners, a Friday ritual nobody keeps past the second quarter. But discipline is a tax on the people least able to pay it. What if the wiki could tell you when it had drifted from the truth — and let the tools you already trust help close the gap?

✶
CHAPTER II

Two readers,
one page

Aveline is built on a small, almost stubborn idea: the same page should read beautifully for a human and address cleanly for a machine. No second system, no exported copy that falls out of sync. One source — rendered as warm prose for the person reading it over coffee, and served as structured context to the agent answering a question three milliseconds later.1

For humans
Deploys run from main. Tag a release, and the pipeline ships to production in about six minutes. Roll back by re-tagging the previous SHA.
For agents
GET /v1/page/deploys { branch: "main", trigger: "tag", ttl_min: 6, rollback: "retag" }
authors' note One write. Two reads. The agent never scrapes your prose; it asks the page directly.

Because the structure is first-class, an agent never has to guess at your formatting or scrape your headings. It queries the page the way it would query an API — and the page answers in kind.

A wiki should be a clean read for humans, and an API for agents — written once, true for both.
CHAPTER III

Knowledge that
keeps itself honest

When an agent reads, it can also notice. A function signature that no longer matches the example. A linked service that has been retired. Aveline turns those small contradictions into gentle, reviewable suggestions — never silent edits — so the page stays close to the code without anyone having to remember to look.2

a principle Suggestions, not silent rewrites. The humans always keep the pen.

The result is a living document in the literal sense — one that accrues corrections the way a well-loved book accrues margin notes. It is open source, self-hostable, and yours to fork. We would rather you read the code than take our word for any of this.
